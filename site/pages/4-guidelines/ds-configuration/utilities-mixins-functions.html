Utilities vs. Mixins vs. Functions

Question

In this site, I have mostly been using utility classes, but according to USWDS, which this design system is based on, I could also use Functions or Mixins. What are the differences between those 3? Are there pros and cons? And can you give examples of how each of them would be implemented on this page?

Answer — Differences (short)

  • Utility classes — precompiled, single-purpose CSS classes added in HTML (example: .shadow-3). They emit no Sass at runtime.
  • Mixins (Sass) — blocks you include (@include) that output CSS rules. They accept parameters and can emit multiple declarations for a selector.
  • Functions (Sass) — return a value (color, length, string, tokens) you use inside properties or variables. They do not emit CSS by themselves.

Pros / cons (short)

  • Utilities — Pros: very fast, predictable, no Sass required in markup. Cons: verbose markup, many classes in stylesheet.
  • Mixins — Pros: parameterizable, bundle many properties into a semantic class. Cons: can duplicate CSS if included many times (larger compiled CSS).
  • Functions — Pros: DRY values, useful for computed tokens and populating CSS variables. Cons: require a Sass build step and are indirect in markup.

When to pick

  • Utilities for fast prototyping and repeated small tweaks (shadows, spacing).
  • Functions for tokenized, computed values used in multiple properties or variables.
  • Mixins for component-level rules where you want a semantic class that bundles multiple properties.

Concrete examples (copy into your code)

1) Utility class (already used on this page)

HTML (already on the page):

<div class="square-9 bg-white shadow-3" style="display:inline-block;"></div>

2) Sass mixin

Add to a Sass partial (e.g. _mixins.scss) and import it into your build:

// _mixins.scss @mixin u-shadow($level) { @if $level == 0 { box-shadow: none; } @else if $level == 1 { box-shadow: 0 1px 4px 0 rgba(0,0,0,0.1); } @else if $level == 2 { box-shadow: 0 4px 8px 0 rgba(0,0,0,0.1); } @else if $level == 3 { box-shadow: 0 8px 16px 0 rgba(0,0,0,0.1); } @else if $level == 4 { box-shadow: 0 12px 24px 0 rgba(0,0,0,0.1); } @else { box-shadow: 0 16px 32px 0 rgba(0,0,0,0.1); } }

Use it in a component stylesheet:

.card { background: #fff; border-radius: 4px; padding: 1rem; @include u-shadow(3); }

HTML:

<div class="card">Component using mixin-generated shadow</div>

3) Sass function

Add a function to a partial (e.g. _functions.scss):

// _functions.scss @function shadow-value($level) { @if $level == 0 { @return none; } @if $level == 1 { @return 0 1px 4px 0 rgba(0,0,0,0.1); } @if $level == 2 { @return 0 4px 8px 0 rgba(0,0,0,0.1); } @if $level == 3 { @return 0 8px 16px 0 rgba(0,0,0,0.1); } @if $level == 4 { @return 0 12px 24px 0 rgba(0,0,0,0.1); } @return 0 16px 32px 0 rgba(0,0,0,0.1); }

Use it directly in a rule or to create a CSS variable:

.card--func { box-shadow: shadow-value(2); }
                    :root { --card-shadow: #{shadow-value(2)}; } .card--var { box-shadow: var(--card-shadow); }

Short recommendation

  • Keep using utilities for style-guide examples and quick markup-level changes.
  • Introduce functions for shared shadow tokens so the same value can be used across mixins, variables, or component rules.
  • Use mixins for semantic component classes (cards, modals) that need a bundle of properties including shadows.

If you want, I can generate the _functions.scss and _mixins.scss files and show where to import them into your existing Sass entry.
